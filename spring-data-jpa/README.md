### Spring Cloud Gateway
Spring Cloud Gateway - это реализация паттерна API Gateway, смысл которого в том, чтобы добавить посредника между
клиентом и микросервисами для более удобного взаимодействия. То есть после внедрения специального сервиса-шлюза
пользователи будут посылать ему все запросы, тот в свою очередь будет осуществлять маршрутизацию (с помощью предикатов и
роутов) этих запросов к соответствующим микросервисам. 

Часто микросервисам назначается случайный порт при старте, что затрудняет прямое взаимодействие с ними, эту проблему 
тоже решает Gateway. Или может быть несколько экземпляров одного и того же микросервиса, тогда Gateway осуществляет 
балансировку запросов между этими экземплярами.

В gateway-service можно перенести функциональность по валидации JWT-токена, что будет удобнее, чем осуществлять проверку 
в каждом сервисе отдельно. Или, например, вести логирование запросов. То есть перенести любую логику, которую нет смысла
реализовывать в каждом микросервисе отдельно.

Gateway может добавлять в запросы какие-либо полезные данные (с помощью фильтров), например информацию о пользователе, 
которая будет использована дальше в микросервисе. Добавлять данные можно так же и в ответ.

Если какой-либо сервис не доступен, то Gateway сможет перенаправить запрос в другой аналогичный микросервис, либо быстро
вернуть ответ о том, что сервис временно не доступен.

Таким образом, Spring Cloud Gateway выступает мощным инструментом для построения гибкой, масштабируемой и безопасной 
микросервисной архитектуры. Он упрощает взаимодействие между клиентами и микросервисами, централизованно решает общие 
задачи и повышает устойчивость системы в целом.

### @Bulkhead
Проблема замедления:

Если один сервис вызывает другой и вызываемый сервис начинает "тормозить", то это начинает влиять на быстродействие 
вызывающего сервиса. Потоки вызывающего сервиса начинают блокироваться, что приводит к исчерпанию пула потоков.

Механизм защиты: принцип работы Bulkhead

Bulkhead заимствует идею из кораблестроения: переборки делят корабль на отсеки, чтобы затопление одного не погубило всё
судно. В программировании это означает жёсткое разделение ресурсов между типами операций.

Для каждого типа запросов выделяется отдельный пул потоков. Запросы, не попавшие в пул, ждут в очереди ограниченное 
время (maxWaitDuration). Если время истекло, запрос отклоняется с ошибкой. Проблемы в одном "отсеке" никак не влияют на
другие виды запросов.

* SemaphoreBulkhead - на основе семафора
* FixedThreadPoolBulkhead - на основе фиксированного пула потоков

Основные настройки:
* maxConcurrentCalls - максимальное количество параллельных вызовов метода
* maxWaitDuration - максимальное количество времени, в течение которого ожидается получение свободного потока

Для ThreadPoolBulkhead:
* maxThreadPoolSize - максимальное количество потоков в пуле
* coreThreadPoolSize - количество кор потоков в пуле
* queueCapacity - допустимое количество потоков в очереди
* keepAliveDuration - когда количество потоков превышает количество кор потоков, данная настройка задает их время жизни,
при условии, что они простаивают
* writableStackTraceEnabled - выводить ли стектрейс bulkhead исключения, по умолчанию true, если значение false то при 
ошибке будет выведена одна строка в лог
